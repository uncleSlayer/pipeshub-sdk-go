The body is a sequence of<br/>`text/event-stream` frames using the event vocabulary described<br/>on `SSEEvent`. The exact subset of events emitted depends on<br/>`chatMode` (see the route description for routing rules).<br/><br/>Lifecycle (all event names are sent verbatim on the wire):<br/>- `connected` — `{ "message": "SSE connection established" }`.<br/>  Fired once on connection by the API layer.<br/>- `status` — progress messages from the AI backend. Possible<br/>  `status` sub-values include `started`, `transforming`,<br/>  `searching`, `processing`, `checking_tools`,<br/>  `generating_answer`, `generating`, `analyzing`, `evaluating`,<br/>  `planning`, `executing`, `retrying`, `continuing`, `success`,<br/>  `skipped`, `pending`, `keepalive`, `cascade_error`, and<br/>  backend-defined values that may be added over time.<br/>- `answer_chunk` — incremental token batches with running<br/>  `accumulated` text, accumulated `citations`, and the<br/>  backend-supplied `confidence` (typically null until the final<br/>  chunk).<br/>- `tool_calls` / `tool_call` / `tool_success` / `tool_error` —<br/>  emitted when the model invokes tools (agent chat modes, or the<br/>  non-agent path when SQL / record-fetch tools are configured).<br/>- `tool_result` / `tool_execution_complete` — additional tool<br/>  lifecycle events emitted only on the agent-mode path.<br/>- `restreaming` — emitted when the LLM is restarted with new<br/>  context (e.g. before a citation-verification pass or<br/>  reflection-driven retry).<br/>- `metadata` — `{}` keep-alive emitted by the JSON-streaming<br/>  branch while waiting for the next safe-to-flush chunk.<br/>- `complete` — `{ "conversation": Conversation, "recordsUsed":<br/>  number, "meta": { "requestId": string, "timestamp": string,<br/>  "duration": number, "recordsUsed": number } }`. Fired once<br/>  after the regeneration is persisted; the new bot response<br/>  replaces the previous one in `conversation.messages` at the<br/>  same index. The AI backend's own `complete` frame is consumed<br/>  server-side and is **not** forwarded — clients see only this<br/>  server-defined frame.<br/>- `error` — `{ "error": string, "details"?: string }`. Fired if<br/>  the stream fails; the previous bot response is replaced with an<br/>  error message and the conversation row is marked FAILED before<br/>  close.<br/><br/>Clients should ignore unknown event names rather than treating<br/>them as errors.<br/> |
